# MIMO 申请项目说明

## 项目名称

面向文博场景的多模态 AI 识别、讲解生成与语音合成 Agent 系统

## 项目解决的核心痛点

文博、景区和展陈行业在内容生产中存在以下问题：

1. 文物资料来源分散，图片、网页、历史描述和后台字段需要人工整理。
2. 单件文物往往需要多个版本讲解词，例如专家讲解、儿童科普、导游口播、短视频脚本等，人工撰写成本高。
3. 文本生成后还需要语义审核、事实一致性校验、字数和时长控制。
4. 音频合成、上传对象存储、回写业务后台等步骤重复且容易出错。
5. 当文物数量扩展到数千件时，传统人工流程难以稳定批量交付。

## Agent 核心逻辑

系统将完整业务流程拆分为多个 Agent：

1. 识别 Agent：根据文物图片、已有描述和后台字段抽取文物名称、年代、形制、纹饰、历史背景等结构化信息。
2. 生成 Agent：根据不同游客画像生成多风格讲解内容。
3. 审核 Agent：检查文本是否包含文物名称、是否过短、是否存在 AI 自称、是否具有基本讲解层次。
4. 语音 Agent：调用 TTS 生成音频，统计音频时长，并上传对象存储。
5. 同步 Agent：将文本、音频 URL、审核状态和结构化知识回写业务后台。

## 当前成果

当前工程已经实现一个可运行的端到端 Demo：

- 支持单件文物处理；
- 支持 zip 压缩包批量上传；
- 支持 8 种讲解风格生成；
- 支持 mock TTS 生成可播放音频；
- 支持本地模拟业务回写；
- 支持替换为真实大模型、TTS、OBS 和业务接口。

## Token 使用合理性

该项目的 Token 使用不是一次性问答，而是批量业务链路中的持续推理和生成：

- 每件文物需要进行一次或多次图文理解与结构化抽取；
- 每件文物会生成多种风格讲解词；
- 每种讲解词需要审核、改写和时长控制；
- 多轮 Agent 协作会产生持续上下文与中间结果；
- 批量处理数千件文物时 Token 消耗会自然放大。

因此百万亿 Token 额度可以用于构建大规模文博知识库、提升多模态识别准确率、优化讲解质量、扩大语种和风格覆盖，并进一步沉淀可迁移到景区导览、工业巡检报告、铁路巡检报告等场景的 Agent 架构。
